== Runtime View

This section describes the **dynamic behavior** of the YOVI system. Following arc42 guidelines, we present a representative selection of **architecturally significant scenarios**:

* **Critical use cases**: User registration, gameplay against bot, bot API usage
* **Key interactions**: How building blocks cooperate
* **Error scenarios**: How the system handles failures
* **Operation scenarios**: System startup and shutdown

These scenarios demonstrate how the building blocks defined in Section 5 collaborate at runtime.

=== Runtime Scenario 1: User Registration (In development)

This scenario shows how a new user creates an account in the system. Currently, only username is required, but verification works the same way.

[plantuml,"User Registration Sequence with Proxy",png]
----
@startuml
actor "Human Player" as User
participant "Nginx Reverse Proxy" as Proxy
participant "Web Frontend\n(webapp/)" as Webapp
participant "API Gateway\n(gateway/)" as Gateway
participant "User Service\n(users/)" as Users
database "MongoDB" as DB

User -> Proxy: 1. Fill registration form\n(username, password, email)
Proxy -> Webapp: 2. Forward request
Webapp -> Webapp: 3. Validate input format\n(client-side validation)

Webapp -> Proxy: 4. POST /api/users/register
Proxy -> Gateway: 5. Forward to gateway (HTTP internal)
Gateway -> Users: 6. Forward request to users service

Users -> Users: 7. Hash password (bcrypt)
Users -> Users: 8. Validate uniqueness

alt User already exists
    Users --> Gateway: 9a. 409 Conflict
    Gateway --> Proxy: 10a. Forward error
    Proxy --> Webapp: 11a. Forward
    Webapp --> User: 12a. Display error message
else New user
    Users -> DB: 9b. INSERT new user document
    DB --> Users: 10b. Confirm insertion

    Users -> Users: 11b. Generate JWT token
    Users --> Gateway: 12b. 201 Created
    Gateway --> Proxy: 13b. Forward
    Proxy --> Webapp: 14b. Forward response
    Webapp --> User: 15b. Show success & redirect
end

note right of Users
  **Quality Goals:**
  * Response < 200ms
  * Password never stored in plain text
  * JWT for stateless auth
end note
@enduml
----

**Steps:**

1. User fills registration form in browser.
2. Reverse proxy receives HTTPS request.
3. Proxy forwards request to Web Frontend.
4. Frontend performs validation.
5. Web frontend sends registration request to Proxy → Gateway.
6. Gateway forwards to User Service.
7. User Service hashes password and checks uniqueness.
8. **Alternative 1**: User exists → 409 Conflict sent back through Gateway → Proxy → Webapp → User.
9. **Alternative 2**: New user inserted into DB, JWT generated. Response flows back through Gateway → Proxy → Webapp → User.

=== Runtime Scenario 2: Playthrough of a Game against Two Players

[plantuml,"User VS User Game Diagram with Proxy",png]
----
actor User1
actor User2
actor GUI1 as "GUI (User1)"
actor GUI2 as "GUI (User2)"
participant "Nginx Reverse Proxy" as Proxy
participant "API Gateway" as Backend
database Database as "Database"

== Initialization ==
User1 -> Proxy: Navigate to Game Initialization
Proxy -> Backend: Request multiplayer game

Backend -> Backend: Create game session
Backend -> Database: Store pending game session

User2 -> Proxy: Join game
Proxy -> Backend: Forward join session request
Backend -> Database: Update game session (2 players connected)

Backend --> Proxy: Game is initialized
Proxy --> GUI1: Forward initialization
Proxy --> GUI2: Forward initialization

GUI1 --> User1: Render initialized game
GUI2 --> User2: Render initialized game

== Turns ==
loop Each turn until game ends
    User1 -> GUI1: Perform move
    GUI1 -> Proxy: Send move
    Proxy -> Backend: Forward move
    Backend -> Database: Update game state
    Backend --> Proxy: Updated state
    Proxy --> GUI2: Forward updated state
    GUI2 --> User2: Render updated game state

    User2 -> GUI2: Perform move
    GUI2 -> Proxy: Send move
    Proxy -> Backend: Forward move
    Backend -> Database: Update game state
    Backend --> Proxy: Updated state
    Proxy --> GUI1: Forward updated state
    GUI1 --> User1: Render updated game state
end
----
*Steps Summary:* Same as original, but all external requests go through **Proxy → Gateway**.

=== Runtime Scenario 3: Play Game vs. AI Bot

[plantuml,"Game vs Bot Sequence with Proxy",png]
----
actor "Human Player" as User
participant "Nginx Reverse Proxy" as Proxy
participant "Web Frontend\n(webapp/)" as Webapp
participant "API Gateway\n(gateway/)" as Gateway
participant "User Service\n(users/)" as Users
participant "Game Engine\n(gamey/)" as Gamey
database "MongoDB" as DB

User -> Proxy: 1. Select "Play vs Bot"
Proxy -> Webapp: 2. Forward selection
Webapp -> Proxy: 3. POST /api/game/init
Proxy -> Gateway: 4. Forward to Game Engine

Gamey -> Gamey: 5. Create game state
Gamey --> Gateway: 6. Return gameId, initial state
Gateway --> Proxy: 7. Forward
Proxy --> Webapp: 8. Forward
Webapp --> User: 9. Render empty board

' Game Loop
loop Each turn
    User -> Webapp: Place piece
    Webapp -> Proxy: POST /api/game/move
    Proxy -> Gateway: Forward
    Gateway -> Gamey: Validate & apply move

    alt Invalid move
        Gamey --> Gateway: 400 Bad Request
        Gateway --> Proxy: Forward error
        Proxy --> Webapp: Forward error
        Webapp --> User: Show error
    else Valid move
        Gamey -> Gamey: Apply move
        Gamey -> Gamey: Check win
        alt Player won
            Gamey --> Gateway: Return winner
            Gateway --> Proxy: Forward
            Proxy --> Webapp: Forward
            Webapp --> User: Show victory
            break
        else Game continues
            Gamey -> Gamey: Bot move
            Gamey --> Gateway: Return updated state
            Gateway --> Proxy: Forward
            Proxy --> Webapp: Forward
            Webapp --> User: Show updated board
        end
    end
end
note over Webapp, DB: Finalization: Update match history

Webapp -> Proxy: Send match update
Proxy -> Gateway: Forward update
Gateway -> Users: Forward update
Users -> DB: Persist match data
DB --> Users: Confirm saved
Users --> Gateway: Confirm
Gateway --> Proxy: Confirm
Proxy --> Webapp: Confirm
Webapp -> User: Update local stats
@enduml
----

=== Runtime Scenario 4: Error and Exception Handling

*All scenarios updated to include Nginx Reverse Proxy as first hop for external requests.*

- **Game Engine Unavailable**: User request → Proxy → Gateway → Gamey → error returned through Proxy.
- **Invalid Move**: Same, request flows through Proxy → Gateway → Gamey.
- **Database Connection Lost**: Requests from Webapp still go through Proxy → Gateway → Users → MongoDB.
