== Runtime View

This section describes the **dynamic behavior** of the YOVI system. Following arc42 guidelines, we present a representative selection of **architecturally significant scenarios**:

* **Critical use cases**: User registration, gameplay against bot, bot API usage
* **Key interactions**: How building blocks cooperate
* **Error scenarios**: How the system handles failures
* **Operation scenarios**: System startup and shutdown

These scenarios demonstrate how the building blocks defined in Section 5 collaborate at runtime.

=== Runtime Scenario 1: User Registration

This scenario shows how a new user creates an account in the system. The authentication service validates credentials and delegates user creation to the users service, then generates a JWT token.

[plantuml,"User Registration Sequence",png]
----
@startuml
actor "Human Player" as User
participant "Web Frontend\n(webapp/)" as Webapp
participant "API Gateway\n(gateway/)" as Gateway
participant "Auth Service\n(authentication/)" as Auth
participant "User Service\n(users/)" as Users
database "MongoDB" as DB

User -> Webapp: 1. Fill registration form\n(username, password, repeatPassword, email)
Webapp -> Webapp: 2. Validate input format\n(client-side validation)

Webapp -> Gateway: 3. POST /register\n{username, password, repeatPassword, email}
Gateway -> Auth: 4. Forward to auth service\n(HTTP internal)

Auth -> Auth: 5. Validate payload\n(required fields, password length,\npasswords match)

alt Validation error
    Auth --> Gateway: 6a. 400 Bad Request\n{error: "Password must have at least 4 characters"}
    Gateway --> Webapp: 7a. Forward error
    Webapp --> User: 8a. Display error message
else Valid payload
    Auth -> Users: 6b. POST /createuser\n{username, email, password}
    Users -> Users: 7b. Check uniqueness\n(username/email already exists?)

    alt User already exists
        Users --> Auth: 8c. 400 Conflict\n"The username field is already in the database"
        Auth --> Gateway: 9c. Forward error
        Gateway --> Webapp: 10c. Forward error
        Webapp --> User: 11c. Display error message
    else New user
        Users -> DB: 8b. INSERT new user document
        DB --> Users: 9b. Confirm insertion
        Users --> Auth: 10b. 201 Created\n{user: {id, username, email, createdAt}}

        Auth -> Auth: 11b. Generate JWT token\n(userId, username, email)
        Auth --> Gateway: 12b. 201 Created\n{success, token, user}
        Gateway --> Webapp: 13b. Forward response
        Webapp -> Webapp: 14b. Store token + username\nin localStorage
        Webapp --> User: 15b. Redirect to /home
    end
end

note right of Auth
  **Quality Goals:**
  * Response < 200ms
  * JWT for stateless auth
  * Payload validation before DB call
end note
@enduml
----

**Steps:**

1. User fills registration form in the web frontend
2. Frontend performs basic validation (client-side)
3. Frontend sends registration request to API Gateway at `POST /register`
4. Gateway forwards to Authentication Service
5. Auth service validates payload (required fields, password length, passwords match)
6. **Alternative 1 (validation error)**: Returns 400, UI shows error
7. Auth service calls User Service at `POST /createuser`
8. **Alternative 2 (user exists)**: User Service returns 400, error propagated to UI
9. **Alternative 3 (new user)**: User Service inserts document in MongoDB
10. Auth service generates JWT token and returns 201 with token
11. Frontend stores token and username in localStorage, redirects to `/home`

=== Runtime Scenario 2: Playthrough of a Game against two players

The following diagram shows the flow of a normal playthrough of a game with two players going against each other (planned feature — not yet implemented):

[plantuml,"User VS User Game Diagram",png]
----
actor User1
actor User2
actor GUI1 as "GUI (User1)"
actor GUI2 as "GUI (User2)"
actor Backend
database Database as "Database"

== Initialization ==
User1 -> GUI1: Navigate to Game Initialization
GUI1 -> Backend: Request multiplayer game

Backend -> Backend: Create game session
Backend -> Database: Store pending game session

User2 -> GUI2: Join game
GUI2 -> Backend: Request to join session
Backend -> Database: Update game session (2 players connected)

Backend --> GUI1: Game is initialized
Backend --> GUI2: Game is initialized

GUI1 --> User1: Render initialized game
GUI2 --> User2: Render initialized game

== Turns ==
loop Each turn until game ends
    User1 -> GUI1: Perform move
    GUI1 -> Backend: Send move
    Backend -> Database: Update game state
    Backend --> GUI2: Updated game state
    GUI2 --> User2: Render updated game state

    User2 -> GUI2: Perform move
    GUI2 -> Backend: Send move
    Backend -> Database: Update game state
    Backend --> GUI1: Updated game state
    GUI1 --> User1: Render updated game state
end

== Finalization ==
alt User1 wins
    Backend -> Database: Store game result (User1 victory)
    Backend --> GUI1: Game finished (User1 wins)
    Backend --> GUI2: Game finished (User2 loses)
    GUI1 --> User1: Show victory screen
    GUI2 --> User2: Show defeat screen
else User2 wins
    Backend -> Database: Store game result (User2 victory)
    Backend --> GUI1: Game finished (User1 loses)
    Backend --> GUI2: Game finished (User2 wins)
    GUI1 --> User1: Show defeat screen
    GUI2 --> User2: Show victory screen
end
----

=== Runtime Scenario 3: Play Game vs. AI Bot

This scenario shows a complete game session between a human player and the AI, using the Rust game engine.

[plantuml,"Game vs Bot Sequence",png]
----
@startuml
actor "Human Player" as User
participant "Web Frontend\n(webapp/)" as Webapp
participant "API Gateway\n(gateway/)" as Gateway
participant "Game Engine\n(gamey/)" as Gamey
participant "User Service\n(users/)" as Users
database "MongoDB" as DB

== Game Initialization ==
User -> Webapp: 1. Select difficulty (bot) and board size
Webapp -> Gateway: 2. POST /game/new\n{size: N}
Gateway -> Gamey: 3. Forward to game engine
Gamey -> Gamey: 4. Create new game state\n(empty board, turn = player 0)
Gamey --> Gateway: 5. Return initial YEN
Gateway --> Webapp: 6. Forward response\n{ok: true, yen: {...}}
Webapp --> User: 7. Render empty board

== Game Loop (until game ends) ==
loop Each turn
    User -> Webapp: 8. Place piece on board (click cell)
    Webapp -> Gateway: 9. POST /game/pvb/move\n{yen, bot, row, col}
    Gateway -> Gamey: 10. POST /v1/game/pvb/{bot_id}\n{yen, row, col}

    Gamey -> Gamey: 11. Validate and apply player move
    Gamey -> Gamey: 12. Check win condition

    alt Player won
        Gamey --> Gateway: 13a. {yen, finished: true, winner: "player"}
        Gateway --> Webapp: 14a. Forward
        Webapp --> User: 15a. Navigate to /game/finished (win)
        break
    else Game continues
        Gamey -> Gamey: 13b. Calculate bot move\n(selected strategy)
        Gamey -> Gamey: 14b. Apply bot move
        Gamey -> Gamey: 15b. Check win condition

        alt Bot won
            Gamey --> Gateway: 16c. {yen, finished: true, winner: "bot"}
            Gateway --> Webapp: 17c. Forward
            Webapp --> User: 18c. Navigate to /game/finished (lost)
            break
        else Game continues
            Gamey --> Gateway: 16d. {yen, finished: false}
            Gateway --> Webapp: 17d. Forward
            Webapp --> User: 18d. Render updated board
        end
    end
end

== Game Finalization ==
Webapp -> Gateway: 19. POST /gameresult (via users service)\n{username, opponent, result}
Gateway -> Users: 20. POST /gameresult
Users -> DB: 21. Save game result document
DB --> Users: 22. Confirm
Users --> Gateway: 23. 201 Created
Gateway --> Webapp: 24. Forward
@enduml
----

**Steps Summary:**

*Initialization:*

1. User selects bot difficulty and board size
2. Frontend requests new game session via `POST /game/new`
3. Game engine creates initial YEN state and returns it

*Game Loop:*

1. Player clicks a cell — frontend sends move via `POST /game/pvb/move` with YEN + row/col + bot id
2. Gateway forwards to `POST /v1/game/pvb/{bot_id}` on the game engine
3. Game engine validates player move and checks win condition
4. If player won → game ends, show victory screen
5. If game continues → engine calculates bot move using selected strategy
6. Bot move applied, win condition checked
7. If bot won → game ends, show defeat screen
8. If game continues → updated YEN returned and board re-rendered

*Finalization:*

1. Game result sent to users service via `POST /gameresult`
2. Result stored in MongoDB

**Architectural significance:**
- Shows **Rust engine** in action (move validation, AI calculation)
- Demonstrates **Strategy Pattern** for bot behavior — bot selected by name via `YBotRegistry`
- Illustrates **YEN notation** for game state exchange across all service boundaries
- Shows **complete flow** from UI → gateway → engine → users service → database

=== Runtime Scenario 4: Error and Exception Handling

This scenario shows how the system handles various failure conditions.

==== Scenario 4.1: Game Engine Unavailable

[plantuml,"Game Engine Down",png]
----
@startuml
actor "User" as User
participant "Web Frontend" as Webapp
participant "Gateway" as Gateway
participant "Game Engine" as Gamey

User -> Webapp: 1. Attempt to make move
Webapp -> Gateway: 2. POST /game/pvb/move
Gateway -> Gamey: 3. Forward request

note over Gamey: Gamey service is down\n(container crashed, network issue)

Gateway --> Gateway: 4. Timeout / Connection refused
Gateway --> Webapp: 5. 502 Bad Gateway\n{ok: false, error: "Game server unavailable"}
Webapp --> User: 6. Display error message

User -> Webapp: 7. Click retry
Webapp -> Gateway: 8. Retry request

@enduml
----

==== Scenario 4.2: Invalid Move (Business Rule Violation)

[plantuml,"Invalid Move",png]
----
@startuml
actor "User" as User
participant "Web Frontend" as Webapp
participant "Gateway" as Gateway
participant "Game Engine" as Gamey

User -> Webapp: 1. Click on occupied cell
Webapp -> Gateway: 2. POST /game/pvb/move\n{position: occupied}
Gateway -> Gamey: 3. Forward

Gamey -> Gamey: 4. Validate move
Gamey -> Gamey: 5. Detect invalid move\n(cell already occupied)

Gamey --> Gateway: 6. 400 Bad Request\n{error: "Cell already occupied"}
Gateway --> Webapp: 7. Forward error\n{ok: false, error: ...}
Webapp --> User: 8. Show "Cell not valid" to user
@enduml
----

==== Scenario 4.4: Invalid or Expired JWT Token

[plantuml,"Invalid Token",png]
----
@startuml
actor "User" as User
participant "Web Frontend" as Webapp
participant "Gateway" as Gateway
participant "Auth Service" as Auth

User -> Webapp: 1. Navigate to /home
Webapp -> Gateway: 2. GET /verify\nAuthorization: Bearer <token>
Gateway -> Auth: 3. Forward

Auth -> Auth: 4. Verify JWT signature and expiry

note over Auth: Token expired or invalid

Auth --> Gateway: 5. 401 Unauthorized\n{success: false, error: "Invalid or expired token"}
Gateway --> Webapp: 6. Forward error
Webapp -> Webapp: 7. Clear localStorage\n(remove username + token)
Webapp --> User: 8. Redirect to / (login)
@enduml
